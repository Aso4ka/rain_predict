4.4 Результаты тестирования разработанного сервиса

В рамках проверки разработанного web-сервиса RainPredict было проведено функциональное тестирование серверной и клиентской частей приложения. Целью тестирования являлась проверка корректности запуска сервиса, доступности пользовательского интерфейса, работоспособности механизма загрузки CSV-файлов, получения прогноза осадков на 24 часа вперёд, а также обработки ошибочных действий пользователя.

Тестирование выполнялось в локальной среде. Серверная часть запускалась с использованием ASGI-сервера Uvicorn. Клиентская часть отдавалась тем же FastAPI-приложением как набор статических файлов. Для проверки использовались тестовые CSV-файлы: dry_day.csv, light_rain_day.csv и heavy_rain_day.csv. Данные файлы имитируют три сценария: отсутствие осадков, слабые периодические осадки и устойчивые сильные осадки.

Команда запуска сервиса:

venv\Scripts\python.exe -m uvicorn backend.main:app --host 127.0.0.1 --port 8000

Основные проверяемые компоненты:

1. Главная страница web-приложения.
2. Страница личного кабинета / прогноза.
3. JavaScript-логика загрузки файла и обращения к API.
4. API загрузки CSV и построения прогноза.
5. API получения результата прогноза.
6. API удаления датасетов из пользовательского хранилища.
7. Обработка ошибок ввода данных.

Проверка ошибок ввода данных

Отдельное внимание было уделено обработке некорректных пользовательских действий, так как web-сервис предполагает загрузку пользовательских файлов. При отсутствии валидации входных данных пользователь мог бы загрузить файл неподходящего формата или пустой файл, что привело бы к некорректной работе модели. В реализованной версии предусмотрены следующие проверки.

1. Проверка расширения файла.
При загрузке файла сервер проверяет расширение имени файла. Если пользователь передаёт файл, не имеющий расширения .csv, прогноз не строится. В ответ возвращается сообщение об ошибке:

{"error":"Можно загружать только CSV файлы"}

Данная проверка позволяет предотвратить передачу произвольных файлов в модуль обработки датасета.

2. Проверка пустого CSV-файла.
Если пользователь загружает пустой CSV-файл, библиотека pandas не может извлечь из него столбцы. В этом случае сервер возвращает ошибку HTTP 400. Полученный ответ:

{"detail":"No columns to parse from file"}

Таким образом, ошибка не приводит к аварийному завершению web-сервиса, а возвращается клиенту в виде диагностического сообщения.

3. Проверка отсутствующего файла при удалении.
При попытке удалить датасет, которого нет в пользовательском хранилище, сервер возвращает HTTP 404. Это предотвращает некорректное удаление и показывает, что backend проверяет существование файла перед выполнением операции.

4. Проверка имени файла.
Для операций с файлами используется функция безопасного получения имени файла. Она отбрасывает путь и оставляет только имя файла, что снижает риск обращения к произвольным директориям за пределами хранилища датасетов.

Вывод по результатам тестирования

Результаты тестирования показали, что разработанный web-сервис выполняет основные функции, заявленные в постановке задачи. Пользователь может открыть интерфейс в браузере, загрузить CSV-файл с метеорологическими данными, получить прогноз осадков на ближайшие 24 часа и управлять загруженными датасетами в хранилище. Серверная часть корректно отвечает на запросы API, а клиентская часть отображает результат прогноза в виде таблицы, карточек и визуального индикатора вероятности осадков.

Проверка на трёх тестовых сценариях показала различие результатов модели в зависимости от характера входных данных. Для сухого сценария была получена низкая вероятность осадков, для сценария слабых осадков — средняя вероятность, а для сценария сильных осадков — повышенная вероятность. Это подтверждает, что модель реагирует на изменение метеорологических признаков и может быть использована в составе web-сервиса для оценки вероятности осадков на сутки вперёд.

Проверка ошибочных сценариев показала, что сервис обрабатывает основные ошибки пользовательского ввода: загрузку файла неподходящего формата, загрузку пустого CSV-файла и попытку удаления несуществующего датасета. Следовательно, разработанное приложение обладает базовой устойчивостью к некорректным действиям пользователя.
